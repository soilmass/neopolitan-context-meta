# The Four Health Gates

Per `MAINTENANCE.md`, every skill is healthy only if all four gates
pass. `skill-audit` runs each gate and rolls them up.

## Gate 1: Recency

**Threshold:** Last update less than 6 months ago.

**Check:** `git log --format=%cI -n 1 -- skills/<name>/` returns a
date within 6 months. If the skill's frontmatter has
`metadata.recency_pin: stable` (acknowledged-as-complete), the gate
passes regardless.

**Why:** A skill not touched in 6+ months may be silently broken or
silently obsolete. The signal can't distinguish "stable" from
"abandoned" — the recency pin is the operator's affirmation.

**Failure mode:** Banner reads "Last update: N months ago (threshold:
<6 months)".

## Gate 2: Test pass rate

**Threshold:** Test pass rate above 90%.

**Check:** Run the skill's evaluation suite (per `ARCHITECTURE.md`).
Skills with pass rates below 90% are flagged.

**Why:** A skill that nominally exists but fails its own tests should
not pass health checks. The 90% threshold is calibrated for skills
with reasonably-sized test suites (10+ assertions).

**Status in v0.1.0:** `N/A`. No SKILL.md test infrastructure exists
yet. Deferred per `governance/INDEX.md` §"INTEGRATION-TESTING.md"
(build trigger: 10+ skills with cross-dependencies).

**Failure mode (when implemented):** Banner reads "Test pass rate:
N% (threshold: >90%)".

## Gate 3: Triggering accuracy

**Threshold:** Triggering accuracy above 85%.

**Check:** Run held-out routing-eval prompts (the protocol is in this
skill's sibling reference; both are listed under the parent SKILL.md
Dependencies section). For each prompt that should hit this skill,
count how often it actually does.

**Why:** Below 85% means the skill is competing with siblings for
routing or has a description that doesn't match how operators
actually phrase prompts. Either is a description-level problem.

**Status in v0.1.0:** `N/A`. No held-out routing-eval suite exists
yet. Deferred per library-root `coverage.md` §"Domains Deferred"
under `skill-evaluate` (build trigger: library reaches 25 skills, OR
first description-change regression slips through).

**Failure mode (when implemented):** Banner reads "Triggering
accuracy: N% (threshold: >85%)".

## Gate 4: Description drift

**Threshold:** Description drift below 10%.

**Check:** Asymmetric containment of description tokens in the body.

  drift = |A − B_full| / |A|

  where:

  - **A** = capability-bearing tokens extracted from the frontmatter
    description. Extraction rule: tokenize the description into words;
    drop the "Do NOT use for" anti-trigger block (those tokens are
    anti-triggers, not claimed capabilities); lowercase; drop a small
    stopword list (articles, conjunctions, structural words like
    "stage", "gate", "skill"); apply crude lemmatization (drop trailing
    `-s` / `-es` / `-ing` / `-ed`).
  - **B_full** = the same tokenization applied to the *entire* body
    (every H2 section), not just one canonical section. The description
    summarizes the whole skill, so every capability it claims should
    appear *somewhere* in the body.

  The measure is asymmetric on purpose: drift is "the description
  claims things the body does not deliver", not symmetric set
  difference. A body that goes deeper than the description summarizes
  is fine and should not flag.

  When `|A| = 0` (description has no capability tokens — pathological),
  drift is reported as N/A with reason `"description has no capability
  tokens"`, not 0.

  An earlier draft used Jaccard distance with body tokens scoped to a
  single archetype-canonical section. That formula always produced high
  drift because the body section was small relative to the description.
  The asymmetric containment measure is the corrected version, pinned
  in v0.2.0.

**Why:** Description drift means the LLM matches against a description
that no longer reflects what the skill does. Routing degrades silently.

**Status in v0.1.0:** Procedurally implementable from SKILL.md alone.
Stage 3 of `skill-audit` runs this gate by hand in v0.1.0. A future
`scripts/audit-skill.py` (deferred — see `coverage.md` §"Domains
Deferred") will mechanize the formula above.

**Failure mode:** Banner reads "Description drift: N% (threshold:
<10%)".

## The banner format

Per `MAINTENANCE.md` §"Auto-Warn Mechanism", a flagged skill's
description gets a warning banner prepended:

```
⚠️ Health Check Failing
Test pass rate: 78% (threshold: >90%)
Last update: 8 months ago (threshold: <6 months)
Triggering accuracy: 91% ✓
Description drift: 4% ✓
```

Each line: gate name, current value, threshold, and ✓ if passing or
no checkmark if failing. The banner is generated by `skill-audit`
Stage 5 and applied by humans (no auto-edit in v0.1.0).

The banner is *not a hard block*. The skill still functions. The
warning is loud enough that operators have to consciously continue
using it rather than passively accepting that nothing is wrong.

## When the banner clears

When the failing signals are corrected (a release fixes the test
failures, the description is updated, a touch refreshes the recency
clock), the banner disappears on the next `skill-audit` run.

## Calibration

The thresholds (6 months / 90% / 85% / 10%) are calibrated for
libraries at the size where this matters (10+ skills with
cross-dependencies). At smaller scales, the gates are still useful
but the thresholds may be too lenient.

The thresholds are tuned annually. Changes to thresholds require a
PR to `MAINTENANCE.md` and a re-run of `skill-audit` against the
whole library.
